# OpenClaw Batch Pages

这个 skill 用于 `fivey-can-read-openclaw` 仓库中的批处理文档页任务。
当用户要你一次处理一组页面、同一类页面、同一目录页面或连续完成多页内容工作时，应使用本 skill。

## 适用场景

典型触发语句包括：

- 批量补写这几页
- 把这组页面全部补完
- 连续处理一批文档页，不要做到一半停下来
- 把某个目录下缺失的页面都完成
- 统一修订同类型页面的文风、结构或 SEO

## 目标

在给定任务范围内，连续完成一整批页面或文件，不要把“完成一小批”误判为“整个任务完成”。

## 工作方式

1. 先明确本轮范围

- 识别用户指定的页面集合、目录范围、文件模式或同类型页面集合
- 如果范围可以从仓库上下文中推断，就直接推断，不要为低风险细节反复确认

2. 先读必要上下文

- 优先阅读仓库根目录 `AGENTS.md`
- 如有必要，阅读 `README.md`、`docs/operations.md`、`package.json`
- 如涉及生成逻辑，优先检查 `scripts/` 下相关脚本

3. 连续推进，不要一小批一停

- 完成一部分页面后，不要默认停下来等“继续”
- 只要仍有同范围内明显未处理的页面，就继续处理下一批
- 批处理中可以给出简短进度更新，但不要请求继续确认

4. 保持项目风格一致

- 保持中文表达清晰、自然、低龄友好
- 保持页面结构、导航逻辑、slug 和输出路径与现有站点一致，除非任务明确要求调整
- 不要无故引入新依赖、新框架或大范围重构

5. 优先最小正确改动

- 优先修改 `content/handcrafted/` 中对应页面
- 只有在确有必要时才改动 `scripts/` 中的生成逻辑
- 不要无故改动与当前批次无关的页面

## 验证要求

完成批处理后，按相关性主动运行验证：

- 内容或生成逻辑改动后，至少运行 `npm run build`
- 需要确认生成结果一致性时，再运行 `npm run check`
- 涉及同步来源或源文档结构变化时，按需运行 `npm run sync`
- 若任务明确要求线上检查，再运行 `npm run verify-live`

如果验证失败，应优先继续定位和修复，而不是立即停下等待下一条指令。

## 只有在以下情况才停下

1. 用户指定范围内的对象已经全部完成
2. 遇到真实阻塞，例如权限、凭据、关键输入缺失或外部依赖受限
3. 不同页面批次需要明显不同的处理策略，且该差异会影响质量、风险或方向
4. 继续执行会触发高风险或破坏性操作

## 输出要求

完成后集中汇报：

- 本轮处理了哪些页面或文件
- 做了什么改动
- 运行了哪些验证
- 是否还有同范围内未完成项
- 剩余风险或下一步建议
