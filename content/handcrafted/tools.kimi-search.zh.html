先讲人话版

🌙 Kimi Search 像一位“拿着 Moonshot 参考书的答案小助手”

这页讲的是 Kimi 在 OpenClaw 里的搜索路线。它走的是 Moonshot web search，输出的也是带引用的答案，不只是蓝色链接列表。

你可以把它想成一个安静但很会整理笔记的小助手。你问它问题，它先去找，再把结果收拾成一段能读懂的话。

适合已经在用 Moonshot / Kimi 生态的人 这页最重要的是看懂 Kimi、Moonshot API key、以及答案型搜索的关系 查看官方原文

第一站

🌙 先把它想成“Moonshot 家的资料小助手”

官方一开始就点明了：Kimi 不是另一套完全不同的怪东西，它本质上就是用 Moonshot web search 来帮你回答问题。

📚 答案型搜索

不是只给链接，而是把资料整理成答案。

📝 带引用

答案后面会把来源挂出来，让你知道它不是凭空编的。

🌙 Moonshot 生态

如果你已经在用 Moonshot 相关的 key 和配置，这条路会很顺。

⭐ 先记一句

Kimi 的风格是“搜完直接讲答案”，不是“搜完丢一堆链接给你”。

第二站

🔑 API key 那一步，就像把门卡插进 Moonshot 的柜子里

官方给了两种 key 名字：KIMI_API_KEY 和 MOONSHOT_API_KEY。你可以把它们理解成同一类门卡的两种标签。

KIMI_API_KEY

像直接写给 Kimi 的门卡。

MOONSHOT_API_KEY

像写给 Moonshot 总柜台的门卡。两者都能用。

openclaw configure --section web

像把网页搜索这张桌子打开，然后把 Kimi 放上去。

🧠 环境变量也行

如果你不想写进配置，也可以放到 Gateway 环境里。像把门卡放进口袋，而不是贴在墙上。

第三站

🧩 配置里的 provider，就是在说“网页搜索以后走哪条路”

官方配置块其实很直接：把 web search provider 指向 kimi，就等于让 OpenClaw 以后先叫 Kimi 出手。

provider: \"kimi\"

像在总搜索台挂牌：“网页搜索请走 Kimi 这条路。”

plugins.entries.moonshot

像把 Moonshot 这套能力包先装好。

webSearch.apiKey

像给这个搜索柜子配门卡。没门卡，Kimi 就进不去。

📌 这页的主线

Kimi 不是独立的大系统，只是 Web Search 里的一条 provider 线路。

第四站

🔁 它怎么出答案，和 Gemini / Grok 的感觉很像

官方也直接说了，Kimi 的思路和 Gemini、Grok 类似：先找资料，再把资料合成带引用的回答。

🧲

先找

让 Moonshot web search 去找相关网页。

🧠

再合成

把找到的资料收拢成一个更像答案的东西。

🔗

带引用

告诉你答案是从哪里来的。

🌙

风格更安静

它给人的感觉像一个不吵，但会把笔记整理好的助手。

第五站

🧷 参数还是很少，说明它主打的是简单直接

Kimi Search 没有很多花里胡哨的筛选项。它就是让你把问题丢进去，然后拿到一个带引用的答案。

query

你要问的问题。

count

你想让它带回多少条结果。

🚫 不主打复杂过滤

更细的国家、语言、时效筛选，这里不是主角。

🎈 这一节一句话

Kimi 不是参数多，而是答案整齐。

最后总结

把这页压成一句最好记的话

Kimi Search 是 Moonshot 家的答案型网页搜索，拿到 key 以后就能让 OpenClaw 去网上找资料、再把答案讲清楚。

如果你下一页只想继续看一页，我建议回去对比 /tools/gemini-search 和 /tools/grok-search。它们和 Kimi 很像，但钥匙和后台味道都不一样。
