先讲人话版

Native Codex plugins，是让 Codex 自己的插件在 Codex 线程里原样工作

这页讲的是 Codex 模式下的一条特别通道：OpenClaw 不把 Codex 插件伪装成一堆动态工具，而是让 Codex app-server 自己处理这些 app 和 plugin 调用。

可以把它想成：OpenClaw 负责把任务交到 Codex 房间门口，房间里面的插件由 Codex 原生管家来安排。

前提：Codex harness 已经能工作 插件调用保留在 Codex native transcript 查看官方原文

第一站

它不是“翻译器”，而是“让原住民自己说原来的语言”

不做的事

OpenClaw 不会把 Codex 插件硬翻译成 codex_plugin_* 动态工具。

会做的事

Codex 插件调用留在 Codex 线程里，由 Codex app-server 执行。

为什么这样设计

原生插件通常有自己的权限、app-backed MCP、上下文和执行记录，硬翻译会丢掉细节。

先看哪页

如果 Codex harness 还没跑通，先回到 /plugins/codex-harness。

第二站

Setup 像把两份名单对齐

1

source Codex home

插件必须装在来源 Codex home 里。

2

inventory

OpenClaw 要知道哪些 plugin 和 app 可见。

3

thread config

线程里的 app config 决定本轮能用哪些能力。

4

native transcript

真正调用时，记录仍按 Codex 原生格式保存。

第三站

安全边界：会动东西的动作不能偷偷溜过去

Destructive action

会删除、覆盖、提交、发布的动作，要遵守 Codex 原生的确认和权限策略。

谁负责执行

app-backed MCP 的执行归 Codex app-server 管，OpenClaw 不假装自己是最终执行者。

V1 support boundary

官方把支持范围说清楚，是为了避免大家把实验能力当成永久承诺。

排错方向

先查 inventory 有没有插件，再查 thread app config 有没有打开，最后看 Codex 端日志。
