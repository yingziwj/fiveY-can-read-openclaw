先讲人话版

🏗 Gateway Architecture 这页，其实是在画 OpenClaw 整座城的总地图

如果你一直分不清 gateway、client、node、webchat 到底谁是谁，这页就是总平面图。它不是某个小命令说明，而是整套系统里每个角色住在哪、走什么路。

读懂这一页，后面很多零散文档会突然对上号。

适合想真正理解整体架构的人 先认地图，再看街道细节 查看官方原文

第一站

🏢 Gateway 是总机房，Clients 是前台，Nodes 是现场小分身

官方 overview 一整段，其实就在帮你把这几个角色分开，不让它们全部挤成“一个 OpenClaw”。

🏢 Gateway

像总机房。权限、模型、通道、会话和对外接口，很多核心事情都在这里协调。

🖥 Clients

像前台窗口。CLI、mac app、web admin 都是来和总机房对话的前台。

📱 Nodes

像分散在外面的眼睛耳朵手脚。手机、macOS、Android、无头节点都属于这类。

💬 WebChat

像专门给聊天场景开的前台窗口，本质上还是连着 Gateway 在说话。

第二站

🔌 Connection lifecycle 在讲“前台和总机房怎么握手、怎么保持通路”

不是每个组件都直接各玩各的。它们之间有一条明确的连接生命周期。

连接建立

像前台先拿门票报到，告诉总机房自己是谁、准备做什么。

持续会话

像值班期间保持对讲机一直连着，消息、状态和工具调用都沿这条线走。

Wire protocol

别被这个词吓到，它本质就是“双方说话用什么信封格式”。

🎈 最后记一句

Gateway Architecture 不是介绍新功能，而是在告诉你：OpenClaw 这座城，路是怎么修的。

最后总结

🎈 把 Gateway Architecture 压成一句最好记的话

OpenClaw 不是单块黑盒，而是一座有总机房、有前台、有节点分身的系统。

接着看 /gateway/configuration 或 /platforms/index，会更容易把这张总地图落到实际部署里。
