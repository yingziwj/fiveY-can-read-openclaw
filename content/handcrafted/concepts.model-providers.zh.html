先讲人话版

🧠 Model Providers 这页，其实是在讲：不同模型商店接进来以后，OpenClaw 怎么统一管

这页不是单个 provider 文档，而是站在更高一层讲规则。它在回答：不同 provider 接进来以后，哪些行为是统一的，哪些差异你必须自己记住。

像插件拥有 provider、API key 轮换、内置 provider 目录，这些都属于“总管层”的规则。

适合想把不同 provider 规则一次看顺的人 重点是 provider 行为差异与轮换 查看官方原文

第一站

🏪 Quick rules 不是废话，而是总商场营业规则

你可以把这一页看成总市场的广播：不同店家虽然都卖模型，但不代表每家结账方式、上货方式、轮换规则都一样。

Plugin-owned provider behavior

有些 provider 不是系统内建，而是插件带进来的，这意味着它们连能力边界都可能跟插件一起变化。

Built-in providers

像系统原装带的店家，路径更短、默认约定也更稳定。

OpenAI / Anthropic / Codex

这些小节不是重复百科，而是在举例说明不同 provider 的接入风格和使用习惯。

🎈 一句话

Model providers 这一页讲的是“管理规则”，不是“货架大全”。

第二站

🔄 API key rotation 是“这把钥匙坏了，下一把怎么顶上来”

轮换行为是生产环境里特别重要的一件事，因为模型能不能持续在线，往往不在第一把 key，而在备用钥匙有没有准备好。

为什么要看

因为 provider 不稳定、额度耗尽、单个 key 失效时，系统能不能平稳切换很关键。

你真正该关心的

不是只知道“支持轮换”，而是知道轮换发生时，默认模型和会话会受到什么影响。

插件 provider 额外注意

如果 provider 是插件带来的，轮换和能力发现往往要连着插件行为一起想。

🎈 最后记一句

不同 provider 能被统一管理，但差异不会消失，尤其在轮换和能力边界上。

最后总结

🎈 把 Model Providers 压成一句最好记的话

这一页在教你怎样用同一张总控台管理不同模型商店，而不是把所有商店都当成一模一样。
