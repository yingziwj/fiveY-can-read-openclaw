先讲人话版

🪵 Amazon Bedrock 这页，其实是在讲：AWS 这栋大楼里，OpenClaw 怎么合法拿到模型通行证

Bedrock 和普通 API key provider 最大不同，是它更像企业云大楼里的正式通道。你不只是填一个 key，还可能碰到区域、实例角色、IAM 这类 AWS 规则。

所以这页重点不是单个命令，而是“在 AWS 规矩里把模型访问权接好”。

适合已经在 AWS 环境里的人 重点是角色、区域和自动发现 查看官方原文

第一站

🧾 支持范围和自动发现，都是在先回答“这栋楼里哪些门能进”

不是所有 AWS 上挂着的模型，都代表当前这套接法能直接用。

支持范围

像楼层导览图，先告诉你哪些房间是真开放的，哪些还只是挂着牌子。

自动发现

像让管理员先巡一圈，看当前 AWS 身份到底被允许访问哪些模型。

为什么这步重要

企业云环境里最怕“理论支持，现实没权限”。

🎈 一句话

先确认能进哪些门，再选要找哪位老师。

第二站

🪪 EC2 Instance Roles 在讲“别总手拿钥匙，也可以靠身份卡刷门”

AWS 世界里很常见的一件事，就是不用显式手填长期密钥，而是让机器自己带着被授权的身份。

手填密钥

像你自己把钥匙放进抽屉里，简单直接，但更需要保管。

实例角色

像这台机器本身就有门禁卡，只要卡上权限正确，OpenClaw 不用额外握长期钥匙。

Quick setup（AWS path）

就是把区域、凭据来源和默认 provider 拼成一条最短路。

🎈 最后记一句

Bedrock 的关键不只是 provider 名字，而是 AWS 身份、区域和可见模型一起对上。

最后总结

🎈 把 Bedrock 这页压成一句最好记的话

在 AWS 世界里，模型访问权和机器身份常常绑在一起。先把这件事弄清，接入才会顺。

如果你所在环境已经是 AWS 正式部署，这页值得和认证页一起看。
