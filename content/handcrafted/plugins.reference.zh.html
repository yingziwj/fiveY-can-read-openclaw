先讲人话版

Plugin reference，是从插件元数据自动整理出来的“索引卡盒”

这页不是一步步教程，而是一盒很薄的卡片。卡片来自 extensions/*/package.json 和 openclaw.plugin.json，用来让维护者快速核对插件名字、清单和公开表面有没有对齐。

如果你是第一次写插件，先看 Building plugins；如果你在检查“这个插件到底有没有被系统认出来”，这页才最有用。

生成型参考页 由插件元数据生成 查看官方原文

第一站

它像图书馆卡盒，不像绘本正文

官方原文很短，只告诉你这页由清单生成，并给出重新生成命令。真正的价值在于：它是很多插件卡片的入口，不是讲概念的长教程。

适合谁

维护者、发版前检查者、想确认插件元数据是否进入文档的人。

不适合谁

第一次学习插件 API 的读者。那类读者应该先看 Building plugins 或 SDK overview。

原文命令
pnpm plugins:inventory:gen
一句话

它回答“有哪些插件卡片”，不负责教你“怎么写完整插件”。

第二站

一张插件卡通常在帮你核对三件事

1

包信息

package.json 像外包装，告诉你 npm 包名、版本、文件出口和 OpenClaw 元数据块。

2

manifest

openclaw.plugin.json 像身份证，告诉系统插件 id、配置、能力和 pre-runtime 线索。

3

公开表面

插件有没有工具、频道、provider、skill 或 setup 信息，可以从卡片里快速扫到。

第三站

什么时候应该回来看这页

它最适合做“发版前点名”。如果某个插件页面不对，通常不是手改这页，而是先检查源元数据，再重新生成。

新增插件后

看它是否出现在参考卡盒里，名字和 id 是否一致。

改 manifest 后

看能力、配置、认证提示有没有被生成器读到。

改 package files 后

确认发布包里该带的东西带上了，不该带的源码或测试物没有混进去。

出错线索

如果 reference 没变化，先怀疑生成命令没跑、源文件没改对、或 manifest id 和包信息没有对齐。

最后记住

Reference 是“看结果”的地方，不是“写规则”的地方

要改插件事实，去改 package.json 和 openclaw.plugin.json；要理解字段意义，去看 Plugin manifest；要判断分发方式，去看 Plugin inventory。
