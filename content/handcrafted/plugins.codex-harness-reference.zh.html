一句话

这里是查旋钮的，不是给新手从零开始看的唯一入口。

认证顺序

先找哪把钥匙，要写在 auth.order.openai 里

Codex-backed OpenAI agent turns 推荐使用 canonical openai/gpt-* model refs。订阅登录、Codex profile、API key fallback 的顺序，要让配置清楚表达。

1

订阅优先

ChatGPT/Codex 登录适合多数 app-server 使用场景。

2

API key 备选

可以作为认证 fallback，但不表示换成 PI 或普通 OpenAI Responses。

3

旧 profile 兼容

openai-codex:* 旧资料仍能服务已有安装。

4

新 ref 别写旧形状

新模型引用用 openai/gpt-*，不要继续写 openai-codex/gpt-*。

app-server

默认用插件管理的 app-server，只有特殊部署才改命令

普通 harness startup 不依赖本机 PATH 上的 codex 命令。参考页里的 app-server 字段，主要给自定义二进制、参数、transport、timeout、mode 和 sandbox 策略使用。

managed binary

像插件自带合适版本的发动机，普通用户不必自己找。

custom command

只有本地测试、特殊发布或运维策略需要时，才用 appServer.command 覆盖。

guardian / yolo

app-server mode 应写在 config 里，临时环境变量只适合一次性测试。

timeouts

启动、请求和 discovery 相关超时要按部署环境调，不要用“永远等”的思路盖住问题。

发现和隔离

模型目录可以 live 探测，也可以用 fallback 目录保守启动

Discovery 会去问 Codex app-server 当前账号和 harness 能看到哪些模型。需要避免启动探测时，可以关掉 discovery，改用 fallback catalog。

discovery.enabled

打开时更贴近真实账号；关闭时启动更少依赖外部探测。

environment isolation

Codex app-server 环境应按插件配置隔离，避免把无关 shell 环境和 secret 一股脑带进去。

native plugins

需要让 Codex 原生插件进入 app-server 时，参考页会说明应该从哪个字段接入。

Computer Use

桌面控制相关能力要和权限、Codex 原生插件、OpenClaw 插件边界分开理解。

排查入口

配置页最有用的时候，通常是“为什么没走到我想的那条路”

遇到模型没出现、runtime 没切、认证失败、app-server 起不来、权限行为不对，先用状态命令确认当前实际路线，再回到 reference 查字段。

/status

先确认当前聊天看到的 runtime 是不是 OpenAI Codex。

/codex status

再看 app-server 连接、账号、rate limits、MCP servers 和 skills。

/codex models

确认 live catalog 里到底有什么模型。

openclaw doctor --fix

修复 legacy refs、stale session route pins 和旧配置形状时很有用。

最后记住

Reference 页是“精确查字段”的地图，不是每次都要全读完的故事书

先用主页面判断要不要启 Codex，再用 runtime 页理解边界；当你真的要改认证、app-server、发现、隔离、权限或超时时，再回到这页查旋钮。
