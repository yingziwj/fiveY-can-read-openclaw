先讲人话版

🔄 Updating 这页，其实是在教你“怎么给 OpenClaw 换新零件，同时别把值班室弄乱”

更新不是简单地“装最新版”，而是要分清你现在是哪种安装路线、想走哪条更新车道、要不要先彩排。

所以官方把 openclaw update 放在最前面，是有原因的。它最像懂你当前安装方式的总维修师。

适合所有已安装用户 重点是 update、channel、dry-run、doctor、restart 查看官方原文

第一站

🛠️ openclaw update 就像“会先看你这台机器怎么装的，再决定怎么修”

这也是它比手动 npm / git 更推荐的原因。

openclaw update

像总维修师先识别你是 npm 装的还是 git 装的，再按对应方式升级。

openclaw update --channel beta

像顺手把你以后要走的更新车道也改了。

openclaw update --tag main

像这次临时指定拿一班特定列车的零件。

openclaw update --dry-run

像先把维修工单打印给你看，不真的动扳手。

第二站

🔁 其他更新法也能用，但更像“你自己拿扳手上”

官方没有禁止，只是把它们放在后面。

🚚

重跑安装器

像再叫一次安装车来，尤其适合你本来就是走安装脚本路线。

📦

npm i -g openclaw@latest

像你自己去商店拿最新版零件，自己换上去。

📦

pnpm add -g openclaw@latest

也是自己换零件，只是用的工具箱不同。

📌

为什么不首推

因为它们不如 openclaw update 那么懂你当前装法。

第三站

🤖 Auto-updater 那段，其实是在讲“要不要给值班室装一个自动换零件的小助手”

默认是关的，这点很合理。

✅ stable 自动更新

不是马上冲，而是会先等一段时间，再带点随机抖动分批推进，像错峰换件。

🧪 beta 自动更新

更激进，像每小时都去前台问一句“有新测试件了吗？”

🚫 dev 不自动装

这条轨道太像施工现场，所以官方让你自己手动决定。

update.checkOnStart: false

像关掉开机时的“有新零件提醒小纸条”。

第四站

🩺 更新后那三步，是“体检、重启、再确认灯亮了”

这三条不要省。

openclaw doctor

像维修后先请医生来体检，看配置迁移、DM 策略、网关健康有没有问题。

openclaw gateway restart

像让值班室重新上岗，别拿旧进程硬撑着用新零件。

openclaw health

像最后再摸一把脉，确认这次换件不是只换了个样子。

🎈 收尾一句

更新不是“装完就走”，而是“装完还要验货”。
