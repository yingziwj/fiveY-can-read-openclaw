# AGENTS.md

## 项目定位

这个仓库用于生成并部署 `fivey-can-read-openclaw` 静态文档站。
目标是把 OpenClaw 文档改写成更适合低龄读者理解的中文故事化页面，并输出到 `dist/`，部署到 Cloudflare Pages。

## 关键目录

- `scripts/`：同步、构建、预览、校验脚本
- `content/handcrafted/`：手写或定制内容页面
- `docs/`：运维、部署和维护说明
- `assets/icons/`：品牌图标和视觉资源
- `dist/`：静态生成输出目录，属于重要产物

## 主要工作流

默认按以下顺序理解和处理任务：

1. 先确认任务影响的是内容、生成逻辑、样式、SEO 还是部署流程
2. 优先阅读 `README.md`、`package.json` 和相关 `docs/`
3. 如涉及生成逻辑，优先检查 `scripts/` 下对应脚本
4. 完成修改后，运行相关验证
5. 最终汇报改动、验证结果和剩余风险

## 常用命令

- 安装依赖：`npm install`
- 同步原始文档：`npm run sync`
- 构建站点：`npm run build`
- 本地预览：`npm run dev`
- 检查生成结果：`npm run check`
- 验证线上站点：`npm run verify-live`

## 执行规则

默认连续执行到任务完成、验证完成或遇到真实阻塞。

对低风险、小范围且可合理判断的事项，直接自主决策。
优先选择最小、正确、符合现有项目结构的改动。
不要擅自扩大范围，不要加入与当前任务无关的重构或新功能。

## 决策规则

对于低风险、小范围、可合理判断的事项，直接自主决策。
优先选择最小、正确、符合现有代码风格和架构的改动。
不要擅自扩大范围，不要加入与当前任务无关的重构、优化或新功能。


## 只有在以下情况才询问用户

仅当满足以下任一条件时，才暂停并询问：

1. 需要执行破坏性或不可逆操作
2. 需要更改部署策略、域名、Cloudflare 配置或其他外部系统关键设置
3. 存在多个代价或方向差异明显的方案，需要用户拍板
4. 缺少关键需求信息，继续执行大概率会做错
5. 受到权限、沙箱、凭据或外部系统访问限制，无法继续

## 内容与生成约束

修改内容或生成逻辑时，保持以下原则：

- 保持中文表达清晰、自然、适合低龄读者理解
- 不要无故删除已有页面、导航项、SEO 标签或静态产物
- 保持生成后页面路径、导航顺序与现有站点逻辑一致，除非任务明确要求调整
- 优先复用现有脚本与生成流程，不要随意引入新依赖或新框架
- 若改动会影响 `dist/` 输出结构、同步来源或页面 slug，必须明确验证

## 验证要求

进行代码或内容修改后，至少选择最相关的验证方式：

- 生成逻辑或模板修改：运行 `npm run build`
- 生成产物一致性检查：运行 `npm run check`
- 涉及同步逻辑：按需运行 `npm run sync`
- 涉及本地浏览体验：按需运行 `npm run dev`
- 涉及线上验证：按需运行 `npm run verify-live`

如果验证失败，应优先继续定位和修复，而不是立刻停下等待下一条指令。

## 输出要求

任务结束时，简明说明：

- 做了什么改动
- 运行了哪些验证
- 是否影响 `dist/`、内容结构、部署流程或 SEO
- 还有哪些风险或后续建议

## 额外要求

除非遇到真实阻塞，不要提出“要不要继续”“是否继续执行”“要不要我接着做”之类的问题。
默认连续执行到可交付结果。对于同一批次的内容，应按照重要性从高到低全部搞完所有页面。

## 批处理任务专用规则

对于批量页面、批量内容、批量生成物、批量修订或同类型文件成组处理任务，不要把“完成一小批”误判为“整个任务完成”。

默认行为如下：

1. 先识别本轮任务的范围，例如指定目录、指定页面集合、指定类型页面或用户口头说明的一整批对象
2. 完成一小批后，不要停下等待用户回复，应继续处理下一小批
3. 若任务属于同一类型且仍有明显未处理对象，默认继续推进直到范围内全部完成
4. 批处理过程中可给出简短进度更新，但不要请求“继续”确认

对于这类任务，只有在以下情况才允许中途停下：

1. 已完成用户指定范围内的全部对象
2. 遇到真实阻塞，例如权限、凭据、外部依赖、网络限制或关键输入缺失
3. 发现不同批次需要采用明显不同的处理策略，且该差异会影响质量、成本或方向
4. 继续执行会触发高风险或破坏性操作

对于批处理文档任务，默认完成标准还应包括：

1. 本轮范围内的目标页面或目标文件都已处理
2. 相关构建或校验已运行，或明确说明未运行原因
3. 没有同范围内明显应继续处理但尚未处理的剩余项

## 默认完成标准

只有同时满足以下条件，才视为任务完成：
1. 用户要求的目标已经落地
2. 已完成相关验证，或明确说明无法验证的原因
3. 没有显而易见且必须马上继续处理的遗留问题
